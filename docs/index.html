The problem we solve

Reviews drive revenue and risk in equal measure. The system that handles them must be fast for shoppers, ironclad against abuse, and never let unmoderated content reach the storefront.

For shoppers 🛒

Nine in ten purchases are influenced by ratings & reviews. A slow, fake-flooded, or down review experience translates directly to abandoned carts and lost trust.

Read-heavy, latency-sensitive workload. Edge-cacheable. Public auth. Cursor pagination so the page stays stable as new reviews land.

For the platform 🛡️

Reviews are write-heavy and abuse-prone. Moderation, ingest, and ops need a different system from the one that serves the storefront — different security, different SLAs.

Internal-only writes (moderate / bulk-ingest / recompute) cannot share TCP listeners with public traffic. Network segmentation isn't a nice-to-have.

System architecture

Two physically separated API surfaces — one for shoppers, one for our internal teams. They share data through a single service core, but never share traffic.

Hexagonal: domain → repository (interface) → service → transport. Two http.Server instances on different ports, different middleware stacks, common service core. Hover or tap nodes for details.

Review lifecycle

Every review passes through a controlled state machine. Nothing reaches the storefront until our moderation pipeline (or a human moderator) clears it.

Strict state machine: PENDING → APPROVED | REJECTED. Public list endpoint filters to APPROVED only. Aggregate is recomputed on transition into / out of APPROVED.

The API surface

Two distinct doors into the system — one for the public storefront, one for our internal tooling.

Stdlib net/http mux with Go 1.22 method-prefix patterns. Both servers share the same service core; only the routes and middleware differ.

Site-facing · :8080

Public, edge-cached, customer-auth.

  • GET/v1/products/{id}/reviews
  • GET/v1/products/{id}/ratings/summary
  • GET/v1/reviews/{id}
  • POST/v1/products/{id}/reviews
  • POST/v1/reviews/{id}/helpful
  • POST/v1/reviews/{id}/unhelpful

Internal · :8081

mTLS-gated, moderation, ops, bulk ingest.

  • GET/internal/v1/reviews/pending
  • GET/internal/v1/reviews/{id}
  • PATCH/internal/v1/reviews/{id}/moderation
  • DEL/internal/v1/reviews/{id}
  • POST/internal/v1/reviews/bulk
  • POST/internal/v1/products/{id}/ratings/recompute

Run it in 30 seconds

Zero external deps. One make run boots both servers seeded with sample data; make test runs the full e2e suite.

git clone https://github.com/shubhambakre/omnimart-ratings-reviews
cd omnimart-ratings-reviews
make run         # boots :8080 (site) and :8081 (internal)
make test        # 5 e2e tests, ~0.5s
make curl-demo   # full smoke test (in another terminal)

Submit a review

curl -X POST http://localhost:8080/v1/products/PROD-1/reviews \
  -H 'Content-Type: application/json' \
  -H 'X-Customer-Id: C-42' \
  -H 'Idempotency-Key: my-key-1' \
  -d '{"rating": 5, "title": "Loved it", "body": "Works as advertised."}'
{

Approve a flagged review (internal)

curl -X PATCH http://localhost:8081/internal/v1/reviews/$ID/moderation \
  -H 'X-Api-Key: dev-internal-key' \
  -H 'Content-Type: application/json' \
  -d '{"approve": true, "notes": "false positive"}'

Tech under the hood

Go 1.22+ stdlib net/http log/slog httptest — that's the entire dependency list.

Every other piece — Cassandra, Kafka, mTLS / SPIFFE, Prometheus, ML moderation — is wired in as a labeled seam. Swapping the in-memory repository for Cassandra, for example, is one new package implementing the existing interface; no handler, service, or test changes.

Production roadmap

From hackathon POC to platform service. Each phase is independently shippable.

  1. Persistence

    Cassandra repo (productId-partitioned); Redis for hot rating summaries with TTL.

  2. Async aggregation

    Kafka review.approved consumer; debounced windowed recompute to absorb hot-key writes.

  3. Auth hardening

    JWT validation against the shopper identity service; mTLS via Istio/Linkerd for the internal API.

  4. ML moderation

    Replace word-list with multi-model pipeline: profanity, PII, spam, sentiment, image NSFW.

  5. Observability

    Prometheus metrics, OpenTelemetry traces, RED dashboards, SLO burn alerts.

  6. Anti-abuse

    Per-customer + per-IP rate limits, device-graph fraud signals, verified-purchase enforcement.
