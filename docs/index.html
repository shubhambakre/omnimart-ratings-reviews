Trust Boundaries
Two API Tiers

Two physically-separate http.Server instances on different ports enforce the network boundary at TCP — not a middleware flag. Different middleware stacks, different auth, no accidental cross-over.

The customer-facing store and the moderation back-office are completely separate systems. Unapproved content can never reach a product page — by architecture, not by convention.

Ports & Adapters
Hexagonal Architecture

Each layer depends only on the one below via interfaces. Swap storage by writing one new package — no handler or service changes. cmd/server/main.go is the only place that knows which concrete types are wired.

The system is built in independent layers. Business rules, database, and API surface can each change independently — reducing risk and making upgrades straightforward.

Pluggable Backends
Storage

Select with STORAGE_DRIVER=memory|sqlite. Both implement ReviewRepository and RatingRepository — zero handler or service changes to swap.

Works in-memory for instant startup, or with a real persistent database for demo stability. Either can be swapped without touching business logic.

The memory repo uses sync.RWMutex for read safety and plain Go maps. It's the default for make run and all e2e tests. No setup required.

// internal/repository/memory/memory.go type ReviewRepo struct { mu sync.RWMutex byID map[string]*domain.Review byProduct map[string]map[string]struct{} idempo map[string]string } // Cursor shared via repository.EncodeCursor(rv) // base64(createdAt_nanos|id) — stable under concurrent writes

SQLite uses modernc.org/sqlite — pure Go translation of the C source. No CGO, no system library. WAL mode enables concurrent reads alongside the single writer.

// Tables: reviews, rating_summaries, idempotency_keys CREATE INDEX idx_reviews_product_status ON reviews (product_id, status, created_at DESC); CREATE INDEX idx_reviews_status_created ON reviews (status, created_at ASC); // NewRepos returns both repos sharing one connection rr, rt, err := sqlite.NewRepos("./data/omnimart.db") // Timestamps: Unix nanoseconds (int64) // PRAGMA journal_mode=WAL; SetMaxOpenConns(1)

Evolution Path
POC → Production

The upgrade seams are already designed in. Each swap requires implementing one interface or replacing one wired dependency — no handler or domain changes.

Every architectural decision preserves the ability to scale — from thousands to hundreds of millions of reviews — without rewriting the system.

🏛️
Legible by design
The hexagonal layout maps cleanly to team ownership: domain, platform, API. Each layer has one responsibility and a defined interface boundary.
🔒
The two-tier split is structural
Two TCP ports encode the trust boundary at the network layer. A public route cannot accidentally reach the moderation queue — they're on different listeners.
🚀
Production-minded from day one
Idempotency, cursor pagination, structured logging, graceful shutdown, and upgrade seams are all present. The gap to production is scale and auth — not architecture.
